# WCTI

一个基于八维战术灵魂模型的足球人格测试，答题后匹配真实国家队并生成 AI 战术报告。

纯 vibe coding 项目，从前端到后端到数据引擎到 AI 报告，全程 AI 辅助生成，人只负责提需求、纠偏和拍板。

主要技术栈：Vite + React + Zustand + ECharts + Recharts，部署在 Vercel，AI 调用智谱 glm-4-flash，缓存用 Vercel Redis。
网址:wcti.online

## 数据引擎

`data_engine/` 是整个项目的基石，负责从原始比赛数据中提炼出 48 支国家队的八维战术评分。

### 数据来源

两个渠道，双源融合：

- **StatsBomb 开源事件数据**，通过 `step1_statsbomb_all.py` 调用 `statsbombpy` 拉取世界杯、欧洲杯、美洲杯、非洲杯的男子国家队比赛事件级数据，粒度到每一次铲球、拦截、过人、传球。
- **FBref 比赛日志**，通过 `step1_cdp_scraper.py` 用 Selenium 爬取 48 支球队 2024-2026 年的所有国家队比赛页面，拿到进球、犯规、抢断成功、助攻等汇总指标。

FBref 爬下来的原始 HTML 存在 `raw_html/`，按球队分目录，`step2_data_cleaner.py` 负责解析 BeautifulSoup 提取表格、清洗多级表头、质检（遇到盾牌页或 500 报错页直接删除源文件），输出干净 CSV 到 `clean_data/`。

### 维度计算

`phase3_soul_calculator.py` 是核心计算引擎，8 个维度的算法思路：

| 维度 | 含义 | 计算方式 |
|------|------|----------|
| Dim1_Heritage | 豪门底蕴 vs 黑马 | 80% 荣誉积分（奖杯数 + 稳定性 + 人才储备 + 品牌溢价） + 20% 近期净胜球波动 |
| Dim2_Domination | 前场压制 vs 防守反击 | 通用框架 + 球探修正 |
| Dim3_Hero | 英雄主义 vs 团队至上 | 通用框架 + 球探修正 |
| Dim4_Pragmatic | 实用功利 vs 理想主义 | 通用框架 + 球探修正 |
| Dim5_Control | 绝对控场 vs 机会主义 | 通用框架 + 球探修正 |
| Dim6_Resilience | 逆境韧性 vs 顺境碾压 | 通用框架 + 球探修正 |
| Dim7_Physical | 身体对抗 vs 技术至上 | 70% StatsBomb 身体事件（Clearance/Duel/Interception） vs 技术事件（Dribble/Pass） + 30% FBref 犯规+抢断 vs 助攻 |
| Dim8_Adaptive | 战术多变 vs 体系固化 | 通用框架 + 球探修正 |

计算完成后做归一化，输出 `Team_8D_Soul_Scores.csv` 和 `.json`，同时复制一份到 `frontend/public/` 给前端匹配用。

荣誉积分数据来自 `heritage_rankings.json`，手工整理了 48 支球队的奖杯数、大赛稳定性、球星储备和品牌加成。

### 球探修正

纯数据算出来的维度有些球队明显不对劲，比如乌兹别克斯坦因为胡桑诺夫的存在，英雄主义和身体对抗理应拉高，但数据层面体现不出来。所以加了一层 `SCOUTING_MODIFIERS` 主观修正字典，覆盖了乌兹别克斯坦、库拉索、挪威、科特迪瓦、约旦、伊拉克、佛得角、海地、新西兰、波黑等球队。

### 蒙特卡洛测试与流量黑洞

`phase4_monte_carlo_test.py` 跑了一个 10000 人的蒙特卡洛模拟，批量生成虚拟用户得分（正态分布，均值 50，标准差 15），用余弦相似度匹配球队，统计各队被选中的频次。

理论均匀命中率应该是 2.08%（1/48），但实际跑出来发现有些球队是"流量黑洞"——特征太中庸，大范围截胡了其他球队的匹配机会，导致另一批球队变成"隐形球队"，命中率不到 0.5%。

这就是为什么你看到的 CSV 里维度分值和后端计算出来的原始值不完全一致——为了打破这种马太效应，对部分维度做了人工微调，拉大球队间的特征差异，让每支队都有被选中的可能性。调整原则是尽量保持数据骨架不变，只在边缘维度上做加减法。

## 前端架构

### 答题引擎

以 Zustand 做状态管理，核心流程：

1. **定标阶段**（动态模式）：先跑 5 道校准题，收集用户的游戏、音乐、运动标签，用标签层级树（`TAG_HIERARCHY`）展开父子关系。
2. **出题算法**：从题库中按维度抽题，每个维度保证 15%-30% 的题目来自标签匹配，其余来自通用题库，确保既有针对性又不失随机性。
3. **计分规则**：每个维度独立计分，取均值后做 `50 + average * 5` 的线性映射到 0-100 区间。

两种模式：简单模式（`simple`）直接答固定题库，动态模式（`dynamic`）先定标再出题。

### 匹配算法

用户完成答题后，拿 8 维得分向量和 CSV 中 48 支球队的向量做双重相似度计算：

- 余弦相似度（权重 0.5）：比方向，维度走势像不像。
- 欧氏距离相似度（权重 0.5）：比距离，绝对值差多少。

加权求和后排序，取最高分作为匹配结果。全部 48 支球队的排名数据也保留在 `allMatches` 里，结算页展示前 5。

### 进度持久化

答题进度、得分、标签全部存 localStorage，刷新不丢。下次打开自动检测缓存，如果上次已经出结果就直接清掉重来，如果卡在答题页就恢复进度。

### 多语言

中英文双切，翻译文本在 `src/i18n/translations.js`，题目和球队数据各有中英文两份 JSON，切换语言时动态重映射。

## AI 战术报告

结算页底部独立组件 `TacticalReport`，失败不影响主界面。

请求链路：前端 POST `/api/generate-report` → Vercel Serverless Function → MD5 哈希查 Redis → 命中直接返回 / 未命中调智谱 API → 写入 Redis（30 天 TTL）→ 返回前端。

限流策略用指数退避 + Jitter，基数是 2^n 秒加随机延迟，最多重试 3 次，避免并发请求同时撞墙。

本地开发时 Vercel Serverless Function 不可用，`api/dev-server.js` 用 Node 原生 http 模块起了一个 mock 服务器，内存缓存模拟 Redis。

## 碰到的问题

- **流量黑洞**：蒙特卡洛模拟发现德国、巴西等特征中庸的球队命中率过高，海地、库拉索等边缘球队几乎不出。通过人工微调维度值解决，让 48 支队都有合理出场率。
- **AI 模块耦合**：早期 AI 报告和结算页强耦合，API 失败直接黑屏。拆成独立组件，自己的加载态和错误兜底自己管。
- **429 限流**：智谱免费版并发限制，加了指数退避重试后才稳定。
- **FBref 爬虫反爬**：部分页面爬下来是盾牌页或 500，`step2_data_cleaner.py` 直接检测并删除坏文件，标记为 `deleted`，不影响后续流程。

## 本地跑

```bash
cd frontend
npm install
$env:ZHIPU_API_KEY="你的密钥"
npm run dev
```

`npm run dev` 同时启动 vite 前端和 mock API 服务器。

## 部署

1. Fork 或 push 到 GitHub
2. Vercel 导入项目，根目录设为 `frontend`
3. Environment Variables 加 `ZHIPU_API_KEY`
4. Storage 里创建一个 Redis 数据库并绑定项目
5. Deploy
